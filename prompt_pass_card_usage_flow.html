TU-08 · Virtual Card + QR
1
QR Token = Time-Limited (2 min rotation)
QR payload encodes a short-lived signed JWT (iss: TU, sub: user DID, exp: 2 min). Backend validates token freshness on scan. Prevents QR screenshot reuse attacks.
SECURITY REQTU-DEP OQ-QR-01
2
Full-Screen Mode = Dark + Gold Glow
Full-screen QR uses black bg (#0A0A14) with gold box-shadow glow — maximizes scanner contrast while maintaining TU brand aesthetics in point-of-sale contexts.
TU CI ACCENT
3
Auto-Close Timer in App Bar
Virtual card auto-closes after 2 minutes of no activity, re-requiring biometric to reopen. Timer shown in app bar: "Auto-closes in MM:SS".
SECURITY REQ
4
QR Payload = Signed VP (not raw user data)
QR encodes a Verifiable Presentation signed by user DID — not raw PII. Partner scans and verifies VP signature against user's DID Document on ShareRing/registry.
SHARERING-DEP OQ10
5

TU-09 · Partner QR Scan
1
App is Passive — Partner Device Scans
The user simply holds their phone screen to the partner's QR reader. The app does not actively push data — it only displays the QR. No Bluetooth, no NFC required for basic flow.
UX PATTERN
2
Verification Flow: Partner → TU Backend → Response
Partner's device decodes QR → extracts VP → verifies user DID signature locally → calls TU membership API to confirm active status → grants benefit. PP backend receives webhook callback.
TU-DEP OQ-VERIFY-01
3
VP Signature Step (Step 2) = Client-Side Only
VP signature can be verified by any party with the user's public DID document — no server roundtrip needed. Enables offline-capable verification in venues with poor connectivity.
OFFLINE CAPABLE
4
qr_scanned Event Logged on Backend
qr_scanned event fired by partner device callback: user_did, partner_id, location_id, token_id, scan_timestamp. Cross-referenced with qr_displayed event for audit trail integrity.

TU-11 · Event Log
1
Event Types — Full Coverage
Four event types captured across the card usage flow: card_opened (TU-07), qr_displayed (TU-08), qr_scanned (TU-09), benefit_granted / benefit_denied (TU-10). All events have a unique EVT-ID for cross-referencing.
EVENT-LOG
2
User View = Human-Readable Receipt Log
User sees plain-language entries grouped by event type (Benefits / Opens / Errors), color-coded Green/Navy/Gold/Red. Enough detail to understand what happened, without raw technical data.
UX PATTERN
3
Admin View = Dark Terminal + Stats
Admin/Partner portal uses dark (#0D0D1A) with monospace log table. Top stats row shows granted/denied/total-opens at a glance. Export to CSV for offline reporting.
ADMIN UX
4
Event Payload Fields
All events carry: event_type, event_id, user_did, partner_id, timestamp_utc, device_os, token_id (where applicable), failure_reason (denied only). Stored on PP backend — not in ShareRing Vault.
DATA SCHEMA
5
Retention Policy = 90 Days (PDPA)
Event logs retained 90 days per PDPA data minimization principle. User can request deletion of their event history. Admins cannot delete individual event records — only bulk expiry applies.
PDPA REQ
6

Event logs stored on PP backend or ShareRing backend or TU system? Recommended: PP backend owns the log (authoritative source), with optional webhook push to TU for their own analytics. Needs decision.
